# PRD: 모바일/공용망 공유 업로드 및 동일 결과 보장

- 작성일: 2026-02-23
- 문서 버전: v1.0 (Draft -> Dev Ready)
- 대상 제품: 신성텍 부적합 현황 보고서 (`index.html`)
- 관련 서버: `scripts/start-local-server.py`, `start-local-server.bat`

## 1. 문서 목적
이 문서는 다음 요구를 제품 요구사항으로 확정하기 위한 기준 문서다.

1. 모바일과 회사 공용 네트워크(사내 LAN)에서 동일 URL로 열람 가능해야 한다.
2. 어느 기기에서 엑셀 업로드를 하든, 다른 기기에서도 동일한 분석 결과를 확인할 수 있어야 한다.
3. 결과 일관성을 보장하는 데이터 동기화 정책, API 계약, 수용 기준을 명확히 정의한다.

## 2. 배경 및 현재 상태

### 2.1 현재 동작(기준 시점: 2026-02-23)
1. HTTP 서버(`start-local-server.py`)는 정적 파일 제공 + API 제공 중이다.
2. 보고서 공유 관련 API가 이미 존재한다.
   - `GET /api/data/cache`
   - `POST /api/data/cache`
   - `GET /api/sw-attachments/manifest`
   - `POST /api/sw-attachments/upload`
   - `POST /api/sw-attachments/delete`
3. 프론트는 업로드 후 로컬 캐시 + 서버 캐시 저장을 시도한다.
4. 다만 운영 기준(버전 정책, 충돌 정책, 일관성 표시, 실패 시 UX)이 명시적으로 정의되어 있지 않다.

### 2.2 문제
1. 사용자 입장에서 "지금 보는 데이터가 최신인지" 즉시 판단하기 어렵다.
2. 다중 기기 동시 사용 시 최종 데이터 기준이 명확하지 않다.
3. 모바일 환경에서 동기화 성공/실패 상태가 운영 기준으로 관리되지 않는다.

## 3. 목표(Goals)
1. **공유 열람성**: 사내 LAN 내 PC/모바일에서 동일 주소로 접속 가능.
2. **공유 일관성**: 업로드 후 모든 단말에서 동일 결과(핵심 KPI/테이블/차트) 확인 가능.
3. **운영 가시성**: 데이터 버전, 업로드 시각, 동기화 상태를 화면에서 명확히 확인 가능.

## 4. 비목표(Non-Goals)
1. 인터넷 외부 공개 서비스화(공인 도메인/공인 인증서).
2. 계정 기반 권한 체계(SSO/LDAP/OAuth) 1차 도입.
3. 실시간 협업 편집(동시 업로드 병합/3-way merge).

## 5. 사용자 및 주요 시나리오

### 5.1 사용자
1. 품질 담당자: 엑셀 업로드 및 최신 데이터 관리.
2. 관리자/현장 리더: 모바일 조회.
3. 유관 부서: 사내 PC 조회.

### 5.2 핵심 시나리오
1. PC-A에서 엑셀 업로드 -> 30초 이내 모바일-B에서 동일 KPI 확인.
2. 모바일-B에서 업로드 -> PC-C에서 새로고침 없이 자동 감지 후 동일 결과 반영.
3. 서버 일시 장애 -> 안내 배지 표시 -> 서버 복구 후 최신 버전 자동 복귀.

## 6. 기능 요구사항 (Functional Requirements)

### FR-01. 접속성
1. 시스템은 사내 LAN URL(`http://<서버IP>:8787/index.html`)로 PC/모바일 접근을 지원해야 한다.
2. 모바일 브라우저(iOS Safari, Android Chrome)에서 주요 화면을 사용 가능해야 한다.

### FR-02. 공유 업로드
1. 사용자가 업로드한 엑셀 분석 결과를 서버 공유 캐시에 저장해야 한다.
2. 저장 시 다음 메타를 포함해야 한다.
   - `version`(정수, 증가)
   - `updatedAt`(UTC ISO-8601)
   - `filename`
   - `fingerprint`(데이터 식별자)
   - `source.kind = uploaded_excel_synced`

### FR-03. 최신본 우선 로딩
1. 클라이언트는 초기 진입 시 서버 캐시를 우선 조회해야 한다.
2. 서버 캐시가 유효하면 로컬 캐시보다 우선 적용해야 한다.
3. 서버 캐시가 없거나 오류일 때만 로컬 캐시 또는 기본 데이터로 폴백해야 한다.

### FR-04. 동일 결과 보장 규칙
1. 같은 `version`과 `fingerprint`이면 KPI/테이블/차트 값이 단말 간 동일해야 한다.
2. 헤더에 `데이터 버전`과 `업데이트 시각`을 표시해야 한다.
3. 단말이 구버전인 경우 자동 또는 수동 갱신 액션을 제공해야 한다.

### FR-05. 동기화 감시
1. 클라이언트는 기본 30초 간격으로 서버 캐시 최신 여부를 확인해야 한다.
2. 신규 버전 감지 시 사용자에게 상태를 보여주고 데이터를 갱신해야 한다.
3. 동기화 실패 시 오류 상태를 배지로 표시하고 재시도해야 한다.

### FR-06. 충돌 정책
1. 동시 업로드는 `Last Write Wins`를 적용한다.
2. 최신 기준은 `version` 우선, 동률 시 `updatedAt` 최신값으로 한다.
3. 버전 증가가 불가능한 경우 업로드를 실패 처리하고 사용자에게 오류를 표시한다.

### FR-07. 사외 첨부 일관성
1. 첨부파일 메타데이터는 서버 manifest를 기준으로 단말 간 동일하게 조회되어야 한다.
2. 첨부 업로드/삭제 후 30초 내 타 단말 반영되어야 한다.

### FR-08. 상태 가시성
1. UI는 최소 아래 상태를 표시해야 한다.
   - `서버 연결 정상`
   - `동기화 중`
   - `오프라인/연결 실패`
   - `최신 상태`
2. 사용자 액션(업로드/동기화) 결과는 성공/실패 토스트 또는 배너로 표시해야 한다.

## 7. 비기능 요구사항 (NFR)

### NFR-01. 성능
1. 10MB 엑셀 업로드 후 공유 저장 완료까지 8초 이내(사내망 기준).
2. 최신 스냅샷 조회 후 첫 렌더 완료 3초 이내(중간 규모 데이터 기준).

### NFR-02. 신뢰성
1. 서버 재시작 후 마지막 정상 캐시 파일을 복원 가능해야 한다.
2. 캐시 저장은 원자적으로 처리해야 한다(임시파일 후 교체).

### NFR-03. 안정성
1. 캐시 저장 API는 페이로드 크기 제한을 가져야 한다(기본 20MB 유지).
2. 첨부 업로드 API는 파일 크기 제한을 가져야 한다(기본 30MB 유지).

### NFR-04. 호환성
1. Windows Chrome/Edge 최신 2개 메이저 지원.
2. iOS Safari/Android Chrome 최근 2개 메이저 지원.

### NFR-05. 보안(사내망 운영 범위)
1. 기본 운영은 사내망 접근으로 제한한다.
2. 업로드 파일/JSON 입력 검증을 수행한다.
3. 최소 감사 로그(시각, 클라이언트 IP, 파일명, 결과)를 남긴다.

## 9. API 계약 (MVP)

### 9.1 `GET /api/data/cache`
- 목적: 최신 공유 보고서 스냅샷 조회
- 응답
  - 200: 캐시 JSON 본문
  - 404: 캐시 없음
  - 500: 서버 읽기 실패

### 9.2 `POST /api/data/cache`
- 목적: 최신 공유 보고서 스냅샷 저장
- 요청: 캐시 JSON 전체
- 검증 규칙
  1. JSON object 여부
  2. payload 크기 제한(<= 20MB)
  3. `version`, `updatedAt`, `snRows`, `swRows` 필드 유효성
- 응답
  - 200: `{ "ok": true, "version": <number> }`
  - 400: 입력 오류
  - 409: 버전 충돌(구버전 저장 시도)
  - 500: 저장 실패

### 9.3 첨부 API
기존 계약 유지:
- `GET /api/sw-attachments/manifest`
- `POST /api/sw-attachments/upload`
- `POST /api/sw-attachments/delete`

## 10. UX/화면 요구사항 (MVP)
1. 헤더에 `데이터 버전`, `업데이트 시각`, `데이터 소스`를 고정 표시.
2. 동기화 상태 배지는 `info/ok/warn/error` 4단계로 통일.
3. 모바일에서 다음 우선 노출:
   - KPI 카드
   - 주요 차트(사내/사외 핵심 2~3개)
   - 최신 업데이트 시각
4. 데이터 불일치 가능 상태(구버전 감지) 시 "최신 데이터 반영" CTA 노출.

## 11. 운영 요구사항
1. 서버는 `start-local-server.bat`로 기동 가능해야 한다.
2. 서버 시작 시 LAN URL 출력 및 QR 제공(가능 시) 유지.
3. 방화벽/포트(8787) 안내 문서를 유지해야 한다.
4. 서버 PC 장애 시 복구 절차 문서를 운영자가 즉시 수행 가능해야 한다.

## 12. 수용 기준 (Acceptance Criteria)

### AC-01 공유 일관성
1. PC-A 업로드 후 30초 이내 모바일-B/PC-C에서 같은 `version` 확인.
2. 세 단말의 KPI(사내 불량, 사내 불량율, 사외 부적합)가 완전히 동일.

### AC-02 충돌 처리
1. 동시 업로드 시 최신 버전 1건만 최종 반영.
2. 구버전 저장 요청은 409로 거절.

### AC-03 장애 복구
1. 서버 재시작 후 마지막 캐시 재조회 가능.
2. 서버 미접속 상태에서 UI에 명확한 경고 표시.

### AC-04 모바일 품질
1. iOS/Android에서 핵심 화면 레이아웃 깨짐 없이 조회 가능.
2. 모바일에서 최신 버전 메타(버전/시각) 확인 가능.

### AC-05 첨부 동기화
1. 첨부 업로드/삭제 후 타 단말에서 30초 내 반영.
